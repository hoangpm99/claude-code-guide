# Requirement Engineering

When starting a project, it's important to understand and write down exactly what the requirements are. Doing that well takes a skill called requirement engineering.

- A requirement is a statement about how a system should behave in order to meet its goals.
- Requirements focus on what the end-user wants, not on the solution or the implementation. Focus on the **WHAT**, not the **HOW**.

## Why requirements matter

- Requirements are the foundation for both development and testing.
- Any flaw in the requirements ends up baked into the code.
- If a requirement can be read in more than one way, developers will fill in the gaps themselves — and end up out of sync.

## Types of requirements

### Functional

*What must the system be able to do?*

E.g. the system must predict whether a patient is having a heart attack.

### Non-functional

*What qualities must the system meet?*

E.g. the system must return a prediction within 1 minute. The system must keep its false-negative rate below 1%.

### Constraints

*What limits are placed on the system?*

E.g. the system must be compatible with the computer's operating system. The system must use a decision tree model for its predictions.
